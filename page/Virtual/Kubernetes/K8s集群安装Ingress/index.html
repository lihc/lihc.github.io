K8s集群安装Ingress

Nginxinc-Ingress

GitHub项目地址：Nginxinc/kubernetes-ingress

Nginx.docs文档：Install Nginx Ingress

本文档介绍了如何使用Kubernetes清单在您的Kubernetes集群中安装NGINX Ingress Controller。

安装先决条件

  1. 确保有权访问Ingress控制器映像：

    • 对于NGINX入口控制器，使用所述图像nginx/nginx-ingress从DockerHub。

    • 对于NGINX Plus Ingress控制器，请按照此处的说明构建您自己的映像并将其推送到私有Docker注册表。

  2. 克隆Ingress控制器存储库并更改到Deployments文件夹：

    git clone https://github.com/nginxinc/kubernetes-ingress/ 
cd kubernetes-ingress/deployments
git checkout v1.9.0

1. 配置RBAC

  1. 为Ingress控制器创建一个名称空间和一个服务帐户：

    kubectl apply -f common/ns-and-sa.yaml
    #common/ns-and-sa.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: nginx-ingress
---
apiVersion: v1
kind: ServiceAccount
metadata:
  name: nginx-ingress
  namespace: nginx-ingress

  2. 为服务帐户创建集群角色和集群角色绑定：

    kubectl apply -f rbac/rbac.yaml
    #rbac/rbac.yaml
kind: ClusterRole
apiVersion: rbac.authorization.k8s.io/v1
metadata:
  name: nginx-ingress
rules:
- apiGroups:
  - ""
  resources:
  - services
  - endpoints
  verbs:
  - get
  - list
  - watch
- apiGroups:
  - ""
  resources:
  - secrets
  verbs:
  - get
  - list
  - watch
- apiGroups:
  - ""
  resources:
  - configmaps
  verbs:
  - get
  - list
  - watch
  - update
  - create
- apiGroups:
  - ""
  resources:
  - pods
  verbs:
  - list
  - watch
- apiGroups:
  - ""
  resources:
  - events
  verbs:
  - create
  - patch
  - list
- apiGroups:
  - networking.k8s.io
  resources:
  - ingresses
  verbs:
  - list
  - watch
  - get
- apiGroups:
  - networking.k8s.io
  resources:
  - ingresses/status
  verbs:
  - update
- apiGroups:
  - k8s.nginx.org
  resources:
  - virtualservers
  - virtualserverroutes
  - globalconfigurations
  - transportservers
  - policies
  verbs:
  - list
  - watch
  - get
- apiGroups:
  - k8s.nginx.org
  resources:
  - virtualservers/status
  - virtualserverroutes/status
  verbs:
  - update
- apiGroups:
  - networking.k8s.io
  resources:
  - ingressclasses
  verbs:
  - get

  name: nginx-ingress

- kind: ServiceAccount
  name: nginx-ingress
  namespace: nginx-ingress

  kind: ClusterRole
  name: nginx-ingress
  apiGroup: rbac.authorization.k8s.io

  3. （仅适用于App Protect）创建App Protect角色和角色绑定：

    kubectl apply -f rbac/ap-rbac.yaml
    #rbac/ap-rbac.yaml
kind: ClusterRole
apiVersion: rbac.authorization.k8s.io/v1
metadata:
  name: nginx-ingress-app-protect
rules:
- apiGroups: 
  - appprotect.f5.com
  resources: 
  - appolicies
  - aplogconfs
  verbs: 
  - "get" 
  - "watch" 
  - "list"
---
kind: ClusterRoleBinding
apiVersion: rbac.authorization.k8s.io/v1
metadata:
  name: nginx-ingress-app-protect

- kind: ServiceAccount
  name: nginx-ingress
  namespace: nginx-ingress
roleRef:
  kind: ClusterRole
  name: nginx-ingress-app-protect
  apiGroup: rbac.authorization.k8s.io

注意：要执行此步骤，您必须是集群管理员。请遵循您的Kubernetes平台的文档来配置管理员访问权限。对于GKE，请参阅基于角色的访问控制文档。

3. 部署Ingress Controller

两个用于部署Ingress控制器的选项：

  • Deployment。如果您计划动态更改Ingress控制器副本的数量，请使用Deployment。
  • DaemonSet。使用DaemonSet在每个节点或节点子集上部署Ingress控制器。

在创建Deployment或Daemonset资源之前，请确保根据您的要求更新相应清单文件中Ingress Controller容器的 命令行参数。

运行Ingress Controller

  • 使用Deployment：默认情况下，当您通过部署运行Ingress Controller时，Kubernetes将创建一个Ingress Controller Pod。

    For NGINX.

    kubectl apply -f deployment/nginx-ingress.yaml
    #deployment/nginx-ingress.yaml

apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-ingress
  namespace: nginx-ingress
spec:
  replicas: 1
  selector:
    matchLabels:
      app: nginx-ingress
  template:
    metadata:
      labels:
        app: nginx-ingress
     #annotations:
       #prometheus.io/scrape: "true"
       #prometheus.io/port: "9113"
    spec:
      serviceAccountName: nginx-ingress
      containers:
      - image: nginx/nginx-ingress:1.9.0
        imagePullPolicy: IfNotPresent
        name: nginx-ingress
        ports:
        - name: http
          containerPort: 80
        - name: https
          containerPort: 443
        - name: readiness-port
          containerPort: 8081

         #containerPort: 9113
        readinessProbe:
          httpGet:
            path: /nginx-ready
            port: readiness-port
          periodSeconds: 1
        securityContext:
          allowPrivilegeEscalation: true

          capabilities:
            drop:
            - ALL
            add:
            - NET_BIND_SERVICE
        env:

          valueFrom:
            fieldRef:
              fieldPath: metadata.namespace
        - name: POD_NAME

$ kubectl get pods --namespace=nginx-ingress
NAME                  READY   STATUS    RESTARTS   AGE
nginx-ingress-5cb2k   1/1     Running   0          3m
nginx-ingress-nhcd2   1/1     Running   0          3m

4. 访问Ingress Contoller

使用DaemonSet创建，则Ingress控制器容器的端口80和443将映射到容器正在运行的节点的相同端口。要访问Ingress控制器，请使用这些端口和正在运行Ingress控制器的群集的任何节点的IP地址。

或者使用Deployment创建，则下面是用于访问Ingress控制器容器的两个选项。

为Ingress Controller Pod创建服务

  • 使用NodePort服务。

    创建一个类型为NodePort的服务：

    kubectl create -f service/nodeport.yaml
    #service/nodeport.yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx-ingress
  namespace: nginx-ingress
spec:
  type: NodePort 
  ports:
  - port: 80
    targetPort: 80
    protocol: TCP
    name: http
  - port: 443
    targetPort: 443
    protocol: TCP
    name: https
  selector:
    app: nginx-ingress

    Kubernetes将在集群的每个节点上随机分配两个端口。要访问Ingress控制器，请使用集群中任何节点的IP地址以及两个分配的端口。

    在Kubernetes文档中阅读有关NodePort类型的更多信息。

  • 使用LoadBalancer服务：

    1. 使用清单为您的云提供商创建服务：

      对于GCP或Azure，运行：

      $ kubectl apply -f service/loadbalancer.yaml

      对于AWS，运行：

      $ kubectl apply -f service/loadbalancer-aws-elb.yaml

      Kubernetes将在TCP模式下分配经典负载均衡器（ELB），并启用PROXY协议以传递客户端的信息（IP地址和端口）。必须将NGINX配置为使用PROXY协议：

      将以下密钥添加到nginx-config.yaml步骤2中的配置映射文件中：

      proxy-protocol: "True"
real-ip-header: "proxy_protocol"
set-real-ip-from: "0.0.0.0/0"

      更新配置图：

      kubectl apply -f common/nginx-config.yaml

      注意：对于AWS，可以使用有关分配的负载均衡器的其他选项，例如负载均衡器的类型和SSL终止。阅读Kubernetes文档以了解更多信息。

      Kubernetes将分配并配置一个云负载均衡器，以对Ingress控制器容器进行负载均衡。

    2. 使用负载均衡器的公共IP访问Ingress控制器。要获取公共IP：

      对于GCP或Azure，运行：

      kubectl get svc nginx-ingress --namespace=nginx-ingress

      对于AWS ELB，公共IP不会由报告kubectl，因为ELB IP地址不是静态的。通常，您应该依靠ELB DNS名称而不是ELB IP地址。但是，出于测试目的，您可以使用获取ELB的DNS名称，然后运行以查找关联的IP地址：kubectl describe nslookup

      kubectl describe svc nginx-ingress --namespace=nginx-ingress

      您可以使用nslookup以下方式将DNS名称解析为IP地址：

      nslookup <dns-name>

      可以以入口资源的状态报告公共IP。有关更多详细信息，请参见报告资源状态文档。

    在Kubernetes文档中了解有关类型LoadBalancer的更多信息。

5. 卸载Ingress Contoller

  1. 删除nginx-ingress名称空间以卸载Ingress控制器以及所有已创建的辅助资源：

    $ kubectl delete namespace nginx-ingress

  2. 删除在该步骤中创建的ClusterRole和ClusterRoleBinding：

    $ kubectl delete clusterrole nginx-ingress
$ kubectl delete clusterrolebinding nginx-ingress
